# 智能体平台参考

公司智能体编排平台的能力速查与编排范式，供撰写新智能体时查阅，免去重复截图研究。

> 本文基于某一时点的平台配置面板整理，平台迭代后以实际面板为准。内容与具体技能无关，示例均用占位。

## 一、基本信息配置

智能体的顶层配置项：

| 项 | 约束 / 说明 |
|----|------------|
| 智能体名称 | ≤64 字符 |
| 智能体描述 | ≤256 字符，是平台判断触发的依据，关键词写全 |
| 应用调用频率限制 | 格式如 `5/1s`（每秒最多 5 次），单位 s（秒）/ m（分）/ h（时）/ d（天），≤20 字符 |
| 开场白 | ≤500 字符 |
| 文件上传 | 开关 + 齿轮设置（见第五节） |
| 用户输入 | 起始收集的字段表（见「开始 / 用户输入」节点） |
| 接口传参 | 参数 / 默认值 / 必填 |
| 语音输入 | 开关 |
| 语音播放 | 开关 |

## 二、变量与引用语法

引用语法**括号内带空格**，按来源分三种前缀：

| 类别 | 前缀 | 示例 |
|------|------|------|
| 内置 / 系统全局变量 | `global.` | `{{ global.time }}`、`{{ global.history_context }}`、`{{ global.chat_id }}` |
| 自建全局变量 | `全局变量.` | `{{ 全局变量.my_state }}` |
| 节点输出 | `节点名.` | `{{ 开始.question }}`、`{{ 条件.branch_name }}`、`{{ 参数抽取.result }}` |

要点：

- 用户输入组件产生的字段（如单选框）挂在 `global.` 下，用 `{{ global.<字段参数名> }}` 引用。
- 自建全局变量在「全局变量配置」里建：变量名 / 类型（string 等）/ 默认值。
- `{{ global.chat_id }}` 存在说明全局变量按会话（chat）持久，可作跨消息状态载体，多轮编排可靠。
- 跨节点传值优先先写入全局变量形成「变量总线」，而不是依赖节点间直连。

## 三、节点能力速查

节点面板分四个 tab：**基础组件 / MCP / 工具库 / 智能体**。基础组件下「业务逻辑」组：多路召回、条件、意图识别、全局变量、自定义函数、指定回复、问题优化、变量赋值、变量聚合、变量拆分、参数抽取，另有开始、循环 / 循环开始、AI 对话。

逐节点（输入 / 配置项 / 输出 / 坑）：

### 开始 / 用户输入

- 全局变量：`time`、`history_context`、`chat_id`，加上「用户输入」里定义的字段。
- 输出：`question`（用户本轮文本消息）、`document`（上传的文件对象）。
- 「用户输入」字段配置项：参数、显示名称、参数提示说明、是否必填、组件类型、密文显示、最小长度、最大长度、显示默认值、默认值。
- 组件类型 7 种：文本框 / 单选框 / 多选框 / 选项卡 / 日期 / 开关 / 标签输入框。
- 接口传参：参数 / 默认值 / 必填。
- 坑：用户输入/文件上传是起始表单，**不能放进循环体**（循环内无法等表单提交），只能置于循环外。

### AI 对话

- 模型（Alibaba Cloud Bailian，可「+ 添加大模型」）：

  ```text
  energy-glm-5.1
  energy-deepseek-v3
  energy-qwen3-30b-a3b-thinking-2507
  energy-qwen3-30b-a3b-instruct-2507
  energy-Deepseek-v3.1
  energy-qwen-plus
  energy-deepseek-r1-distill-qwen-14b
  ```

- 配置项：AI 模型（必填）、系统角色、提示词、历史聊天记录（范围 节点 / 工作流，N 轮）、MCP 开关、Skill 开关、开启思考、返回内容。
- 输出：`answer`、`reasoning_content`。
- 坑：
  - 无法强制调用某个挂载的 skill，只能靠提示词约束。
  - 文件路径不能作结构化输出带出节点，产物信息只能经 `answer` 文本往返。
  - 需要「产出 → 后处理串联」（如生成文件再发布下载链接）时，两个 skill 要挂在**同一个** AI 对话节点，最终结果写进该节点的 `answer`。
  - 判定节点靠「历史聊天记录=工作流、N 轮」读到用户上一句。

### 参数抽取

- 输入变量：下拉选（全局变量 / 上游节点）。
- 每个参数配：名称、字段 key、类型（`string` / `num` / `json`）、描述。
- 输出：`result`。

### 条件

- IF：变量 / 条件 / 值，可多条件（且 / 或）。
- 操作符：为空、不为空、包含、不包含、等于、大于等于、大于……
- 支持 ELSE 与「添加分支」多路。
- 输出：`branch_name`。
- 坑：**每个分支（含 ELSE）都必须接下游节点，不能悬空**。「结束本轮」也要落成一个终端节点（如指定回复，返回内容=开），不能靠留空实现。

### 循环 / 循环开始

- 循环类型：固定次数 / 数组循环 / 变量次数。**固定次数上限 9**。
- 「循环开始」子节点输出：`loop_item`、`loop_index`；循环节点输出：`loop_result`。
- 坑：**无 while（条件退出）、不支持 break**。用固定次数（≤9）当重试上限，靠「条件命中后把分支连到循环容器之外的节点」当退出；次数用尽循环自然结束，正常出口也要接到下一步以免卡死。

### 指定回复

- 回复类型：引用 / 自定义（自定义=直接写 Markdown，可引用变量，平台按 Markdown 渲染）。
- 配置项：返回内容开关（是否把回复发给用户）。
- 输出：`answer`（回复内容）。
- 坑：
  - **返回内容=关 时不产生终端回复**：若它是路径最后一个节点，前端等不到本轮回复、会话挂起不结束（实测）。作路径收尾时必须开返回内容。
  - 节点内容是**字面回复文本**，写「结束当前会话并等待用户下一条消息」之类的指令没有任何控制语义——结束靠路径自然走完，不靠文字。

### 变量赋值

- 选一个变量 ← 引用变量 或 自定义值；类型 string 等。是变量总线的写入端。
- **只能引用单个变量，不能拼接多个**：需要多源合成时，先让 AI 对话/参数抽取产出单一输出，再整段赋值。

### 全局变量（配置）

- 变量名 / 类型 / 默认值。

### 其他可用节点

意图识别、问题优化、变量聚合、变量拆分、多路召回、自定义函数，以及多模态解析节点（见第五节）。

## 四、多轮与状态

- `history_context`、`chat_id` 与 AI 对话的「历史聊天记录」共同支撑多轮：模型能读到此前对话。
- 全局变量按 `chat_id` 持久，可作跨消息的 `stage` 状态载体。

## 五、文件上传与多模态解析

文件上传设置：单次上传最多文件数、每个文件最大 MB、上传的文件类型。四类文件各需一个对应解析节点，把 `{{ 开始.document }}` 转成可用文本 / 结构：

| 类型 | 扩展名 | 解析节点（输出字段） |
|------|--------|----------|
| 文档 | TXT / MD / DOC / DOCX / HTML / CSV / XLSX / XLS / PDF | 文档提取（`content`）；PDF 另可选「PDF文档解析」（`markdown`，需选 `文档解析模型`，版式/表格保真更好） |
| 图片 | JPG / JPEG / PNG / GIF | 视觉模型 |
| 音频 | MP3 / WAV / OGG / ACC / M4A | 语音转文本 |
| 视频 | MP4 / AVI / MKV / MOV / FLV / WMV | 视频理解 |

- 「文档提取」输入 `选择文档 = {{ 开始.document }}`，覆盖全部文档类型（含 PDF）；「PDF文档解析」输入 `选择文件 = {{ 开始.document }}`，仅 PDF、输出 Markdown，是并列的可选高保真路径。
- 上传物落在 `{{ 开始.document }}`，必须先经上述节点解析，再喂给下游 AI 对话。

## 六、可复用编排范式

### 起始收参

用户输入 / 文件上传置于循环体外（表单不能进循环）：单选框做枚举选择、文件上传做素材导入、文本框收主题。字段值经 `{{ global.* }}` / `{{ 开始.* }}` 引用。

### 变量总线

跨节点传值先用「变量赋值」写入自建全局变量（`{{ 全局变量.* }}`），下游再引用，避免依赖节点直连的脆弱连线。

### 多轮对话与确认门

**首选：会话式单节点 + 标记门**。把多轮澄清/修改放进一个 AI 对话节点（开「历史记录=工作流」），用户发消息即一次迭代、次数不限；节点在对话里呈现卡片、逐轮推进，只有判定到用户的「结束意图」时才在回复末尾打一个标记（如 `[[READY]]`），由一个条件放行到下游的确定性节点（渲染 / 提交 / 发布）。

```text
开始
  → AI 对话[主对话](历史=工作流；返回内容=开；挂所需 Skill)
       逐轮推进业务；仅当用户明确结束时，回复末尾单独一行输出 [[READY]] + 下游要消费的数据
  → 条件({{ 主对话.answer }} 包含 [[READY]]?)
       是 → AI 对话[执行](确定性动作，从 {{ 主对话.answer }} 取数据；返回内容=开) → 本轮结束
       否 → 指定回复(自定义一句简短引导语；返回内容=开) → 本轮结束，等用户下一条消息重跑进主对话
```

优点：无循环、迭代不限、执行确定化（对话再多轮也不会误触发）。唯一前提「每条用户消息会从开始重跑工作流并进主对话」是聊天智能体标准行为。

「结束」不是节点：平台没有结束节点，条件分支也不能悬空，一轮会话以路径自然走完收尾，且**末节点必须「返回内容=开」**——否则前端等不到终端回复、会话挂起（实测：否分支接返回内容=关的指定回复即挂起）。

**次选：循环节点**。确认门放进循环，固定次数 ≤9 作重试上限，确认分支连到循环外当退出。⚠️ 隐患：无 break 跳不出、上限仅 9、且依赖「循环体逐圈停等输入」这一未验证行为——仅在验证过停等且迭代次数可控时用。

**兜底：状态机**。不用循环，每条消息从开始重跑，用自建全局变量记 `stage` 分流；依赖 `chat_id` 持久，可靠但节点偏多。

判定要点：「是否结束/确认」交给模型判断，别用条件对原话硬匹配关键词——「确认一下把页数改成 20」含「确认」却是修改，会误判。

### 文档解析链

```text
文件上传 → 文档提取({{ 开始.document }}) → 变量赋值(落全局变量) → 下游 AI 对话消费
```

按文件类型分流（如 PDF 走高保真 Markdown 解析）时，用一个条件节点分出多路，各路解析后统一落同一全局变量，下游只认该变量（变量总线，避免多分支直连合流）：

```text
文件上传 → 条件(按扩展名分流)
  ├ PDF   → PDF文档解析({{ 开始.document }}) → 变量赋值(落 g_source ← markdown)
  └ 其他文档 → 文档提取({{ 开始.document }})   → 变量赋值(落 g_source ← content)
→ 下游读 {{ 全局变量.g_source }} 消费
```

### Skill 调用与产物回传

AI 对话挂 Skill 执行确定性工作；产物路径不能带出节点。要传给下游或给用户看的数据（大纲 JSON、下载链接、文件地址等）只能随 `answer` 文本走：会话式方案里下游节点从 `{{ 主对话.answer }}` 提取（约定放回复末尾、独立围栏便于稳定提取）；若产出与发布必须连做（生成文件后立即发链接），则挂在**同一** AI 对话节点、结果写进该节点 `answer`。

## 七、硬约束与坑清单

- 用户输入 / 表单不能进循环体。
- 循环无 while、不支持 break，只能固定次数 + 条件分支连出循环当退出。
- AI 对话无法强制调 skill（提示词约束）；文件路径不能结构化带出节点，只能经 `answer` 文本往返。
- 多轮判定用模型（CONFIRM/REVISE），不要用条件对原话硬匹配关键词。
- 系统全局 `global.` 与自建全局 `全局变量.` 前缀不同，写错取不到值；引用括号内要带空格。
- 条件分支（含 ELSE）必须接下游节点，不能悬空。
- 每条路径的末节点必须「返回内容=开」，否则本轮会话不结束（前端一直等待）；指定回复里写「结束会话」之类字面文本没有控制语义。

## 八、待平台实测确认项

撰写具体智能体时，在画布调试里花两分钟确认：

- 循环体是否逐圈停等用户输入（决定用方案 A 还是 B）。
- string 全局变量的容量上限（大段 JSON 文本能否装下；超限就压缩或分存）。
- 发布类 skill 的入参是文件路径还是文件对象。
- 单选框「选项」的录入面板位置与格式（组件类型选单选框后的选项编辑区）。
- 「条件」能否读到 `{{ 开始.document }}` 的文件名/扩展名（用「包含 .pdf」按类型分流的前提；否则单节点「文档提取」兜底）。
- AI 对话「返回内容=关」时，其 `answer` 是否仍计入「历史聊天记录=工作流」（当前方案不依赖，仅备查）。
- 多分支能否汇入同一下游节点（多入边合流）：**实测 AI 对话节点无法从多个分支汇入**，需按分支各自复制一份下游节点（或拆成串联的二选一条件）；其他节点类型是否可合流仍待逐一确认。
