# create-ppt 重构：模板复用 + WPS 式两道确认门

> 技能已由 `create-chem-ppt` 重命名为 `create-ppt`（企业汇报/培训通用定位），文件夹 `skills/create-ppt/`。
> §1–4 的克隆回填内核已实现落地；本轮新增：第 5 套「通用模板」、起始「用户输入」表单（单选模板 + 文件上传）、两条入口分支、分页卡片预览、平台编排改为**会话式主对话 + 标记门**（弃用循环）。实测两点定论：「文档提取」已覆盖 PDF（取消 PDF 专用分支）；平台分支无法汇入同一节点，故主对话按「上传/无上传」复制两份同构节点，各带一套就绪门 + 渲染。

## Context

目标是让智能体达到 WPS AI 的体验：**用户输入（选模板 + 可上传文档）→ 确认意图 → 生成大纲 → 确认 → 执行生成**。最初两个根本缺口已解决：

1. **产出不像模板（已解决）。** 模板是完整设计稿：每页由自由形状、装饰图片、文本框拼成，几乎不用占位符。旧 `build_pptx.py` 在空白版式上**程序化画色块**，模板设计被整个丢弃。现改为**克隆模板设计页 + 文本回填**，产出与模板视觉一致。
2. **没有确认门（已解决）。** 现按两道确认门推进，禁止从一句话直奔渲染。

已验证的关键结论：把模板页**深拷贝 + rId 重映射**克隆到同一 package 内、再回填文本，图片/主题/字体全部完好，未引用 media 在 save 时自动丢弃（通用模板 40MB → 产物约 19MB）。

本轮新增决策：
- **第 5 套「通用模板」**：来自 `assets/PPT制作培训与模板.pptx`（95 页讲解+模板混排稿），精修出 55 页可复用集（封面 1 + 目录 1 + 章节 4 + 内容 48 + 封底 1），供任意主题兜底。
- **起始表单**：对话开始用「用户输入」组件收集 单选模板（4 专用 + 通用）+ 文件上传（仅文档，不支持 ppt 作模板）+ 主题文本框。
- **模板以单选为准**；推断意图与单选冲突时向用户确认是否切换。
- **两条入口**：上传文档/粘贴成段文稿/大纲 → 跳过意图门直奔大纲门；仅一句话主题 → 走意图门。
- **大纲确认改分页 Markdown 卡片**（图三风格），取代扁平单行。
- **平台编排改为会话式主对话 + 标记门**（弃用循环）：主对话多轮推进两道门，判定到结束意图时吐 `[[READY]]`，由条件放行到独立渲染节点。因平台分支无法汇入同一节点，主对话按「上传/无上传」复制两份同构节点，各带一套就绪门 + 渲染。规避循环「跳不出 / 上限 9 / 停等未验证」的隐患。

目标产物：技能自身在任意环境可跑（对话内两道确认门），并给出平台工作流编排方案还原截图体验。

## 一、技术底座：克隆回填替代程序化绘制

### 已验证的克隆内核

在同一个 `Presentation` 对象内操作（源页与目标页同 package，无需跨包搬 media）：

```python
R = "{http://schemas.openxmlformats.org/officeDocument/2006/relationships}"

def clone(prs, i):
    src = prs.slides[i]
    new = prs.slides.add_slide(src.slide_layout)
    for sh in list(new.shapes):                      # 清掉新页自带占位符
        sh._element.getparent().remove(sh._element)
    mapping = {}
    for el in src.shapes._spTree:
        if el.tag.endswith(('}nvGrpSpPr', '}grpSpPr')):
            continue
        cp = copy.deepcopy(el)
        for node in cp.iter():                       # r:embed / r:id 重映射到新页 rels
            for attr, old in list(node.attrib.items()):
                if attr.startswith(R):
                    if old not in mapping:
                        rel = src.part.rels[old]
                        mapping[old] = (new.part.relate_to(rel.target_ref, rel.reltype, is_external=True)
                                        if rel.is_external else
                                        new.part.relate_to(rel.target_part, rel.reltype))
                    node.set(attr, mapping[old])
        new.shapes._spTree.append(cp)
    return new
```

流程：打开模板 → 按大纲克隆所需页（可重复引用同一源页）→ 删除全部原始页 → 回填文本 → 保存。

**踩过的坑（勿走回头路）**：不要用 `rels._add_relationship(reltype, target, rid)` 保留原 rId——与新页自带的 layout rel 冲突，`save()` 时在 `target_ref` 断言崩溃。必须重映射 rId。

### 槽位定位

模板里 shape name 大量重复（`任意多边形: 形状 85` 出现多次），**必须用 shape id 定位**，不能用 name。组合（GROUP）内的文本框需递归遍历。

**通用模板的精修**：原始稿 95 页是「PPT制作规范与模板」讲解+版式混排。用 `index_template.py` 出草稿后，脚本化精修（`scripts` 外的一次性工具）保留可复用页、丢弃规范讲解页：
- 封面 `i=0`、目录 `i=1`、封底 `i=94` 直接留用；
- `01./02./03./04.` 四张数字分隔页（`i=2,4,15,22`，原判 content）重标为 `section`：数字槽转 `no`（按章节序号自动改写、支持 >4 章循环取号）、小标题槽转 `title`；
- 内容页只取 `i∈[23,92]` 且 `2≤items≤6` 的 48 张（过密页与规范讲解页排除）。
最终 55 页（封面 1 / 目录 1 / 章节 4 / 内容 48 / 封底 1）。注意：`index` 里的 `i` 必须保留原始页序（`build_pptx` 按它 `clone_slide(prs, i)`），精修只删条目不重排号。

### 页型索引 `templates/index/<模板>.json`

每个模板页 = 一个「版式样例」，索引它的页型与文本槽位：

```jsonc
{
  "template": "员工安全知识培训",
  "source": "assets/员工安全知识培训.pptx",
  "slides": [
    { "i": 0, "kind": "cover",
      "slots": [ {"sid": 7, "role": "title", "cap": 14},
                 {"sid": 5, "role": "subtitle", "cap": 10},
                 {"sid": 9, "role": "decor"} ] },
    { "i": 1, "kind": "toc", "items": 4, "slots": [...] },
    { "i": 2, "kind": "section", "slots": [ {"sid": 3, "role": "no", "auto": "序号"},
                                            {"sid": 4, "role": "title", "cap": 12} ] },
    { "i": 5, "kind": "content", "items": 4,
      "slots": [ {"sid": 12, "role": "item_head", "idx": 0, "cap": 6},
                 {"sid": 13, "role": "item_body", "idx": 0, "cap": 40}, ... ] }
  ]
}
```

- `kind`：`cover | toc | section | content | table | warning | closing`
- `role`：`title | subtitle | item_head | item_body | no | decor`
- `decor`（`01`、`PART 01`、`PROJECT APPLICATION`、`LOGO`）**保留原样不回填**；`no` 类按章节序号自动改写。
- `cap`：字符容量，取自原占位文本长度并按角色收敛，作为写作硬约束（防溢出，这是模板不崩版的关键）。
- `items`：该页可容纳的要点数——**大纲反过来适配模板**，只生成模板存在的 items 数，避免运行时删形状破坏设计。

索引由 `scripts/index_template.py` 自动出草稿（按 z-order/字号/位置/文本长度启发式猜 kind 与 role），再由模型精修一次落盘。**4 个模板共 ~120 页，属一次性离线工作**，运行时不再需要模型读模板。

## 三、交互：入口分流 + 两道确认门

### 入口分流（决定走几道门）

- **直接文本/大纲路径**：用户上传了文档，或粘贴了成段文稿/已有大纲 → **跳过门 ①**，用选定模板生成骨架、以其内容填充优化，直奔门 ②。
- **主题路径**：只给一句话主题 → 走门 ①。
- **模板锁定**：模板一律以用户选定值（起始单选）为准，不由措辞改判。模型仍从主题推断场景；若与选定模板明显不符（选了安全培训、主题却像年终总结），在意图卡追加一行「您选了 X 模板，但主题更像 Y，是否切换？」，等用户确认。

### 门 ① 意图确认卡（仅主题路径）

抽取 5 字段，选用模板填选定值，其余缺失给默认值并**明示**，以卡片复述、等用户确认：

```text
1. 制作目标：动火作业安全培训
2. 目标受众：一线操作工与监护人
3. 选用模板：员工安全知识培训   ← 起始单选，锁定
4. 页数规模：高度概括（12–16 页）

## 五、平台工作流编排（会话式主对话 + 标记门，弃用循环）

**弃用循环的原因**：循环节点无法主动跳出（无 while、不支持 break），固定次数上限仅 9，且「循环体每圈停等用户输入」是未验证的非标准行为——把多轮确认压在循环里，隐患集中。

**改为会话式**：AI 对话节点开「历史聊天记录=工作流」后天然支持多轮，用户发消息就是一次迭代，次数不限。把「意图澄清 + 大纲生成 + 多轮修改」放进**主对话节点**，由它按两道门推进；只有当它判定用户有**结束意图**时，才吐出最终大纲并打一个 `[[READY]]` 标记，由一个**条件**把关、放行到**独立的渲染节点**。渲染因此确定化——平时对话再多轮也不会误渲。

**为何是两份主对话**：实测平台多分支无法汇入同一节点，而「有上传」路径必须先经「文档提取」再进主对话、「无上传」路径直接进主对话——两路无法在主对话处合流，只能各接一份**配置完全一致**的主对话（及各自的就绪门 + 渲染）。代价是系统角色/提示词要在两份间保持同步。

前提（聊天智能体标准行为）：每条用户消息都会从「开始」重跑一遍工作流，主对话节点凭「历史记录」看到完整上下文；`{{ global.chat_id }}` 已证明全局变量按会话持久。

### 5.1 真实变量总线与节点能力（画布截图实证）

变量引用语法（括号内带空格，三种前缀）：
- 内置/系统全局变量：`{{ global.变量名 }}`（如 `{{ global.template }}`、`{{ global.chat_id }}`）
- 自建全局变量：`{{ 全局变量.变量名 }}`（如 `{{ 全局变量.g_source }}`）
- 节点输出：`{{ 节点名.变量名 }}`（如 `{{ 开始.question }}`、`{{ 条件.branch_name }}`）

| 变量 | 来源 | 说明 |
|------|------|------|
| `{{ global.template }}` | 开始节点全局变量（由「用户输入」单选框产生） | **选定模板，权威**。取值为 5 个模板键之一 |
| `{{ 开始.question }}` | 开始节点输出 | 用户本轮文本消息（主题、一段话或修改意见） |
| `{{ 开始.document }}` | 开始节点输出 | 上传的文件对象；统一经「文档提取」节点解析（实测已覆盖 PDF） |
| `{{ global.chat_id }}` | 系统全局变量 | 会话 id，全局变量按会话持久（多轮可靠） |
| `{{ 全局变量.g_source }}` | 自建全局变量（string） | 上传文档解析后的正文文本，由「文档提取」`content` 写入；无上传时为空 |
| `{{ 主对话A.answer }}` / `{{ 主对话B.answer }}` | 节点输出 | 主对话本轮回复（卡片预览 + ```json 块，就绪时带 `[[READY]]`）。**上传/无上传两份主对话各有自己的 answer**，各自下游就绪门与渲染节点只读本分支的 answer |

会话式方案不再需要 `g_intent` / `g_outline` 这类跨节点状态变量——意图与大纲都活在**对话历史**里，最终大纲随主对话的 `answer` 直接交给渲染节点。变量总线只剩 `global.template`、`开始.*`、`g_source`。

节点能力（据配置面板）：

| 节点 | 实证要点 |
|------|---------|
| **开始 / 用户输入** | 用户输入是一张字段表（参数 / 显示名称 / 是否必填 / 组件类型）；本流只需一个字段 `template`（组件类型=单选框）。**文件上传**是节点上的独立开关 + 齿轮设置（限文档类型、单次数量、单文件大小），产出 `{{ 开始.document }}`。这些都在循环体外，不受「表单不能进循环」限制 |
| **文档提取** | 输入 `选择文档 = {{ 开始.document }}`；输出 `content`（解析后正文）。覆盖 TXT/MD/DOC/DOCX/HTML/CSV/XLSX/XLS/PDF 全部文档类型（**实测 PDF 亦可提取，本方案不再单列 PDF文档解析**） |
| **PDF文档解析** | 平台仍提供（PDF 高保真转 Markdown），但**本方案不用**——实测「文档提取」已覆盖 PDF，无需按类型分流 |
| **循环** | 类型：固定次数 / 数组循环 / 变量次数；有「循环开始」子节点（输出 `loop_item`/`loop_index`），循环节点输出 `loop_result`。**固定次数上限 9**；**无 while、不支持 break**。⚠️ 本方案因「跳不出 + 停等未验证」**弃用**，此处仅留作平台事实 |
| **AI 对话** | 可选模型：`energy-deepseek-v3` / `energy-qwen3-30b-a3b-instruct-2507` / `energy-qwen-plus` 等（生成类用前两者，判定类用 qwen-plus 够快）。有 **系统角色 + 提示词** 两个输入框；**历史聊天记录**可设（节点 / 工作流，N 轮）——判定节点靠它读到用户上一句；**Skill** 开关（挂载 create-ppt / download-link-publisher）；输出 `answer` / `reasoning_content`。❌ 无法强制调 skill（提示词约束）；❌ 文件路径不能结构化带出，outline 只能经 `answer` 文本往返；渲染与发链接必须挂**同一个 AI 对话节点** |
| **参数抽取** | 输入变量从下拉选（全局变量 / 开始）；每参数配 名称 / 字段 key / **类型（string/num/json）** / 描述；输出 `result`。类型可选 `json`，但跨节点仍走 `{{ 参数抽取.result }}` 文本 |
| **条件** | IF 变量 / 条件 / 值，操作符：为空 / 不为空 / 包含 / 不包含 / 等于 / 大于等于 / 大于…；支持 ELSE 与「添加分支」；输出 `branch_name` |
| **指定回复** | 回复类型：引用 / 自定义（自定义=直接写 Markdown，可引用变量）；输出 `answer`（回复内容） |
| **变量赋值** | 选一个变量 ← 引用变量 或 自定义值，类型 string 等。**只能引用单个变量、不能拼接多个**——需要多源合成时，先让 AI 对话/参数抽取产出单一输出，再整段赋值 |
| **全局变量（配置）** | 变量名 + 类型（string）+ 默认值；本流只新建 `g_source`（可选 `g_done` 防重渲） |

另可用节点：**意图识别**、**问题优化**、变量聚合 / 变量拆分 / 多路召回 / 自定义函数。本方案把意图与大纲的判定都交给**主对话节点自身**（凭对话历史），不再拆分独立判定节点。

### 5.2 起始节点与全局变量配置

1. **用户输入 → 字段 `template`**：显示名称「模板」、是否必填=开、组件类型=单选框，选项填 5 个模板键：`员工安全知识培训` / `项目立项汇报模版` / `年终安全工作汇报模版` / `重大事件专项汇报` / `通用模板`。产出全局变量 `{{ global.template }}`。
2. **文件上传**：开启开关 → 齿轮设置：只勾「文档」类型（TXT/MD/DOC/DOCX/HTML/CSV/XLSX/XLS/PDF），单次 1–3 个、单文件 ≤50MB。**不支持 ppt 作模板**（模板由单选决定，上传只作内容素材）。产出 `{{ 开始.document }}`。
3. **开场白**：见 SKILL.md `metadata.opening`。
4. **新建全局变量**（变量名 / 类型 string / 默认值留空）：`g_source`（上传文档解析文本）；可选 `g_done`（防止生成后重复渲染）。

### 5.3 拓扑（会话式主对话 + 标记门；分支不合流，复制两份）

```text
[开始]  输出 {{ 开始.question }} {{ 开始.document }}；全局 {{ global.template }} {{ global.chat_id }}；自建全局 {{ 全局变量.g_source }}（默认空）
  │
条件[上传分流]  {{ 开始.document }}（两分支无法汇入同一节点，各自续接一份主对话）
  ├ IF「有上传」 {{ 开始.document }} 不为空
  │      → 文档提取(选择文档={{ 开始.document }}) → 变量赋值 g_source ← {{ 文档提取.content }} → 主对话A
  └ ELSE「无上传」 {{ 开始.document }} 为空 → 主对话B（不写 g_source，保留默认/上一轮值）
       文档提取实测已覆盖 PDF，无需按扩展名分流；主对话只读 {{ 全局变量.g_source }}——变量总线
  │
AI对话[主对话A / 主对话B]  两份配置完全一致（仅所处分支不同）
   Skill: create-ppt；历史记录=工作流（多轮）；返回内容=开
   载荷：选定模板 {{ global.template }} / 上传素材 {{ 全局变量.g_source }} / 用户消息 {{ 开始.question }}
   系统角色管两道门 + 结束意图判定，逐轮推进：
     · g_source 非空 → 视为直接文本/大纲，跳过意图澄清，直接出大纲预览
     · 否则 → 先澄清并复述创作意图 → 用户认可后再出大纲
     · 出大纲：make_outline 骨架 + 按 writing-guide 填充 + --preview 分页卡片，末尾附完整 ```json 块
     · 用户要改 → 重出「卡片 + json」，轮次不限
     · 用户明确要生成（生成吧/就这样/可以了）→ 本轮回复＝最终「卡片 + json」，最末单独一行输出 [[READY]]
  │（每份主对话各接一套就绪门 + 渲染，互不合流）
条件[就绪?A / 就绪?B]  {{ 主对话A.answer }} / {{ 主对话B.answer }} 包含 [[READY]]?
  ├ 是 → AI对话[渲染A / 渲染B]  Skill: create-ppt + download-link-publisher（两份同构）
  │        从本分支 {{ 主对话*.answer }} 提取 json → build_pptx → check_pptx → 发布 → 链接写进 answer
  │        （可选：变量赋值 g_done="yes"）
  └ 否 → 结束（等用户下一条消息，重跑进主对话）
```

要点：

- **无循环**：迭代由用户发消息驱动，次数不限——彻底绕开「跳不出 / ≤9 / 停等」。
- **渲染确定化**：主对话平时不渲染；只有它吐出 `[[READY]]`，条件才放行到独立渲染节点，渲染只调一次，不误渲/重渲。
- **两道门在对话里自然完成**：意图卡、大纲卡片都由主对话写进回复，用户直接聊天修改，比循环里「指定回复 + 判定 + 条件」省一大串节点。
- **json 随 answer 走**：最终大纲的 ```json 块在主对话回复里，渲染节点从 `{{ 主对话.answer }}` 提取即可，不需要 `g_outline` 全局变量。用户会看到卡片下方的 json 块（符合输出契约，WPS 式结构数据，可接受）。
- **直接文本/大纲**：靠 `g_source` 是否为空在主对话**内部**分流，不再单设 direct/topic 两条外部支路。
- **文档提取覆盖 PDF（实测）**：所有文档类型（含 PDF）统一走「文档提取」`content`，取消原「PDF文档解析」分支与按扩展名分流，拓扑更简。
- **分支不合流 → 主对话复制两份**：平台多分支无法汇入同一节点，故「有上传/无上传」各接一份**同构**主对话（及各自的就绪门 + 渲染）。两份的系统角色/提示词/Skill 配置必须保持一致——改一处要同步另一处。上传素材经 `变量赋值` 落 `g_source`，主对话统一读 `{{ 全局变量.g_source }}`（无上传时为空），不出现 `{{ 文档提取.content }}` 这类跨节点直连引用。

### 5.4 节点逐项配置

**条件[上传分流]** — 变量 `{{ 开始.document }}`，两路：
- IF「有上传」：`不为空` → 文档提取 → 变量赋值 g_source → 主对话A。
- ELSE「无上传」：`为空` → 直接进主对话B。

两分支无法汇入同一节点，故各自续接一份主对话（配置见下，两份同构）。

**文档提取** — `选择文档 = {{ 开始.document }}`，输出 `content` → 变量赋值 `g_source ← {{ 文档提取.content }}`。**实测覆盖全部文档类型（含 PDF）**，无需再按扩展名分流、也不再挂「PDF文档解析」。仅在有新上传时运行；无上传的后续轮次 `g_source` 保留上一轮值（空则主对话走意图澄清）。

**AI 对话[主对话A / 主对话B]** — 全流程核心，**两份配置完全一致**（分支不合流才复制，改一处须同步另一处）。模型 `energy-deepseek-v3`（能编排 skill 又能写作）；**Skill 挂 create-ppt**（做 make_outline，不挂 download-link-publisher）；**历史聊天记录＝工作流、10 轮**；返回内容＝开；开启思考＝关。

系统角色：

```text
你是企业 PPT 生成智能体，内置 5 套设计模板，用 create-ppt 技能工作。与用户多轮对话，按两道确认门推进，绝不跳步直奔生成。

【模板】以“选定模板”为准，不改判。若素材/主题场景与选定模板明显不符，提示一行「您选了 X，但更像 Y，是否切换？」（换模板需用户重开一轮重选）。

【门① 意图】
- “上传素材”非空 → 视为直接文本/大纲，跳过意图澄清，直接进入门②。
- 否则 → 先澄清创作意图（制作目标 / 目标受众 / 页数规模 / 补充说明，缺失给默认值并标注），用编号卡片复述，让用户确认或修改；认可后进入门②。

【门② 大纲】
- uv run scripts/make_outline.py 生成骨架，按 references/writing-guide.md 填充（要点条数不增删、每条不超该页 caps），uv run scripts/make_outline.py --preview 输出分页卡片预览。
- 本轮回复固定两段：先分页卡片预览，再一个完整的 ```json 大纲块。
- 用户要改 → 重出同格式，轮次不限。

【结束意图 → 放行渲染】
- 仅当用户明确要生成（生成吧 / 就这样 / 可以了 / 开始做）时，本轮回复＝最终「分页卡片 + 完整 ```json 块」，并在最末单独一行输出：[[READY]]
- 在用户明确要生成之前，绝不输出 [[READY]]，也不渲染 pptx、不发布链接（那是下游渲染节点的事）。

【规范】纯中文、术语规范、每页单一主题、结论先行、不编造数据。所有 uv run 命令不加 timeout 参数。
```

提示词（每轮输入载荷）：

```text
选定模板：{{ global.template }}
上传素材（可能为空）：
{{ 全局变量.g_source }}
用户消息：
{{ 开始.question }}
```

**条件[就绪?A / 就绪?B]** — 每份主对话各接一个：变量 `{{ 主对话A.answer }}` / `{{ 主对话B.answer }}`、操作符「包含」、值 `[[READY]]` → 各自渲染分支；ELSE → 结束（等下一条消息）。

**AI 对话[渲染A / 渲染B]** — 两份同构，各接在对应就绪门后。模型 `energy-deepseek-v3`；Skill 挂 create-ppt + download-link-publisher；历史记录关；返回内容＝开。提示词（把 `{{ 主对话.answer }}` 换成本分支的 `{{ 主对话A.answer }}` / `{{ 主对话B.answer }}`）：

```text
从下面文本中提取 ```json 大纲块，忽略 [[READY]] 标记，不要修改大纲内容，直接渲染。

{{ 主对话.answer }}

用 create-ppt 技能完成：
1. 提取 json 块，原样写入 outline.json
2. uv run scripts/build_pptx.py --outline outline.json --out <主题>.pptx
3. uv run scripts/check_pptx.py --outline outline.json <主题>.pptx 自检；报溢出就改短对应文案重渲
4. 调用 download-link-publisher 发布，回复给出下载链接
```

可选防重渲：渲染成功后接一个「变量赋值 `g_done="yes"`」，并在主对话系统角色加一句「若 `g_done` 已为 yes 且用户未提新需求，视为新一轮、重新澄清意图」。

### 5.5 技能侧契约与待确认项

契约（与对话式使用一致，不引入平台耦合）：outline 阶段固定回复「分页 Markdown 卡片 + 一个完整 ```json 代码块」。AI 对话只有 `answer` 一个文本出口，最终大纲的 json 必须落在主对话的 `answer` 里，渲染节点才能稳定提取。为便于提取，让主对话把 json 放在回复末尾、用独立 ```json 围栏。

> **唯一运行期前提**（聊天智能体标准行为，低风险）：每条用户消息会从「开始」重跑整个工作流并进入主对话节点，主对话凭「历史记录＝工作流」看到完整上下文。搭个最小工作流（`开始 → AI对话(历史=工作流) → 结束`）连发两条消息即可验证：第二条能接着第一条的上下文回答＝成立。
>
> **已实测结论（本轮更新，画布已回填）**：
> 1. 「文档提取」可提取 PDF —— 取消「PDF文档解析」分支与按扩展名分流，全部文档统一走「文档提取」`content`。
> 2. 多分支无法汇入同一节点 —— 「有上传/无上传」两路各复制一份同构主对话（及各自就绪门 + 渲染），不再尝试三路合流。
>
> **剩余待实测项（配置时在画布回填）**：
> 1. 两份主对话「历史记录＝工作流」是否各自独立累积会话历史——同一 chat 内若用户先无上传聊几轮、再补传文档（分支切换），上下文能否延续；若不延续，主对话系统角色需靠 `g_source` + 用户消息重建上下文，而非依赖历史。
> 2. `download-link-publisher` 入参是文件路径还是文件对象（决定渲染节点提示词第 4 步措辞）。
> 3. 用户输入里「组件类型=单选框」后的选项录入面板位置；主对话 json 块偏长时渲染节点能否稳定提取（放末尾独立围栏基本无碍）。
