# create-ppt 重构：模板复用 + WPS 式两道确认门

> 技能已由 `create-chem-ppt` 重命名为 `create-ppt`（企业汇报/培训通用定位），文件夹 `skills/create-ppt/`。

### 5.1 真实变量总线与节点能力（画布截图实证）

变量引用语法（括号内带空格，三种前缀）：
- 内置/系统全局变量：`{{ global.变量名 }}`（如 `{{ global.template }}`、`{{ global.chat_id }}`）
- 自建全局变量：`{{ 全局变量.变量名 }}`（如 `{{ 全局变量.g_source }}`）
- 节点输出：`{{ 节点名.变量名 }}`（如 `{{ 开始.question }}`、`{{ 条件.branch_name }}`）

| 变量 | 来源 | 说明 |
|------|------|------|
| `{{ global.template }}` | 开始节点全局变量（由「用户输入」单选框产生） | **初始选定模板（起始单选默认值）**，取值为 5 个模板键之一。对话内可被用户确认切换覆盖——当前模板以主对话依对话历史解析为准，并写入 outline.json 的 template |
| `{{ 开始.question }}` | 开始节点输出 | 用户本轮文本消息（主题、一段话或修改意见） |
| `{{ 开始.document }}` | 开始节点输出 | 上传的文件对象；统一经「文档提取」节点解析（实测已覆盖 PDF） |
| `{{ global.chat_id }}` | 系统全局变量 | 会话 id，全局变量按会话持久（多轮可靠） |
| `{{ 全局变量.g_source }}` | 自建全局变量（string） | 上传文档解析后的正文文本，由「文档提取」`content` 写入；无上传时为空 |
| `{{ 主对话A.answer }}` / `{{ 主对话B.answer }}` | 节点输出 | 主对话本轮回复（分页卡片预览，就绪时最末单独一行 `[[READY]]`）。**上传/无上传两份主对话各有自己的 answer**，各自下游就绪门与渲染只读本分支的 answer；卡片即大纲的无损序列化，渲染节点用 `--from-preview` 直接重建 outline.json |

会话式方案不再需要 `g_intent` / `g_outline` 这类跨节点状态变量——意图与大纲都活在**对话历史**里，最终大纲随主对话的 `answer` 直接交给渲染节点。变量总线只剩 `global.template`、`开始.*`、`g_source`。

节点能力（据配置面板）：

| 节点 | 实证要点 |
|------|---------|
| **开始 / 用户输入** | 用户输入是一张字段表（参数 / 显示名称 / 是否必填 / 组件类型）；本流只需一个字段 `template`（组件类型=单选框）。**文件上传**是节点上的独立开关 + 齿轮设置（限文档类型、单次数量、单文件大小），产出 `{{ 开始.document }}`。这些都在循环体外，不受「表单不能进循环」限制 |
| **文档提取** | 输入 `选择文档 = {{ 开始.document }}`；输出 `content`（解析后正文）。覆盖 TXT/MD/DOC/DOCX/HTML/CSV/XLSX/XLS/PDF 全部文档类型（**实测 PDF 亦可提取，本方案不再单列 PDF文档解析**） |
| **PDF文档解析** | 平台仍提供（PDF 高保真转 Markdown），但**本方案不用**——实测「文档提取」已覆盖 PDF，无需按类型分流 |
| **循环** | 类型：固定次数 / 数组循环 / 变量次数；有「循环开始」子节点（输出 `loop_item`/`loop_index`），循环节点输出 `loop_result`。**固定次数上限 9**；**无 while、不支持 break**。⚠️ 本方案因「跳不出 + 停等未验证」**弃用**，此处仅留作平台事实 |
| **AI 对话** | 可选模型：`energy-deepseek-v3` / `energy-qwen3-30b-a3b-instruct-2507` / `energy-qwen-plus` 等（生成类用前两者，判定类用 qwen-plus 够快）。有 **系统角色 + 提示词** 两个输入框；**历史聊天记录**可设（节点 / 工作流，N 轮）——判定节点靠它读到用户上一句；**Skill** 开关（挂载 create-ppt / download-link-publisher）；输出 `answer` / `reasoning_content`。❌ 无法强制调 skill（提示词约束）；❌ 文件路径不能结构化带出，outline 只能经 `answer` 文本往返；渲染与发链接必须挂**同一个 AI 对话节点** |
| **参数抽取** | 输入变量从下拉选（全局变量 / 开始）；每参数配 名称 / 字段 key / **类型（string/num/json）** / 描述；输出 `result`。类型可选 `json`，但跨节点仍走 `{{ 参数抽取.result }}` 文本。⚠️ 本方案**不用**——LLM 抽取无法还原卡片里没有的 `src`/`caps`（不可臆造），且长 json 有截断风险；最终大纲由渲染节点从卡片经 `--from-preview` 脚本化重建。此处仅留作平台事实 |
| **条件** | IF 变量 / 条件 / 值，操作符：为空 / 不为空 / 包含 / 不包含 / 等于 / 大于等于 / 大于…；支持 ELSE 与「添加分支」；输出 `branch_name` |
| **指定回复** | 回复类型：引用 / 自定义（自定义=直接写 Markdown，可引用变量）；输出 `answer`（回复内容） |
| **变量赋值** | 选一个变量 ← 引用变量 或 自定义值，类型 string 等。**只能引用单个变量、不能拼接多个**——需要多源合成时，先让 AI 对话/参数抽取产出单一输出，再整段赋值 |
| **全局变量（配置）** | 变量名 + 类型（string）+ 默认值；本流只新建 `g_source`（可选 `g_done` 防重渲） |

另可用节点：**意图识别**、**问题优化**、变量聚合 / 变量拆分 / 多路召回 / 自定义函数。本方案把意图与大纲的判定都交给**主对话节点自身**（凭对话历史），不再拆分独立判定节点。

### 5.2 起始节点与全局变量配置

1. **用户输入 → 字段 `template`**：显示名称「模板」、是否必填=开、组件类型=单选框，选项填 5 个模板键：`员工安全知识培训` / `项目立项汇报模版` / `年终安全工作汇报模版` / `重大事件专项汇报` / `通用模板`。产出全局变量 `{{ global.template }}`。
2. **文件上传**：开启开关 → 齿轮设置：只勾「文档」类型（TXT/MD/DOC/DOCX/HTML/CSV/XLSX/XLS/PDF），单次 1–3 个、单文件 ≤50MB。**不支持 ppt 作模板**（模板由单选决定，上传只作内容素材）。产出 `{{ 开始.document }}`。
3. **开场白**：见 SKILL.md `metadata.opening`。
4. **新建全局变量**（变量名 / 类型 string / 默认值留空）：`g_source`（上传文档解析文本）；可选 `g_done`（防止生成后重复渲染）。

### 5.3 拓扑（会话式主对话 + 标记门；分支不合流，复制两份）

```text
[开始]  输出 {{ 开始.question }} {{ 开始.document }}；全局 {{ global.template }} {{ global.chat_id }}；自建全局 {{ 全局变量.g_source }}（默认空）
  │
条件[上传分流]  {{ 开始.document }}（两分支无法汇入同一节点，各自续接一份主对话）
  ├ IF「有上传」 {{ 开始.document }} 不为空
  │      → 文档提取(选择文档={{ 开始.document }}) → 变量赋值 g_source ← {{ 文档提取.content }} → 主对话A
  └ ELSE「无上传」 {{ 开始.document }} 为空 → 主对话B（不写 g_source，保留默认/上一轮值）
       文档提取实测已覆盖 PDF，无需按扩展名分流；主对话只读 {{ 全局变量.g_source }}——变量总线
  │
AI对话[主对话A / 主对话B]  两份配置完全一致（仅所处分支不同）
   Skill: create-ppt；历史记录=工作流（多轮）；返回内容=开
   载荷：初始模板 {{ global.template }}（当前模板由主对话依历史解析）/ 上传素材 {{ 全局变量.g_source }} / 用户消息 {{ 开始.question }}
   系统角色管两道门 + 结束意图判定，逐轮推进：
     · g_source 非空 → 视为直接文本/大纲，跳过意图澄清，直接出大纲预览
     · 否则 → 先澄清并复述创作意图 → 用户认可后再出大纲
     · 首轮出大纲：make_outline 骨架落盘 outline.json + 按 writing-guide 填充写盘 + --preview 分页卡片，回复只含卡片
     · 用户要改 → 开轮先从历史卡片 --from-preview 重建 outline.json（沙箱每轮全新），--patch 就地改（结构性改动才重跑 make_outline）+ --preview，本轮只回「卡片 + 变更摘要」，轮次不限
     · 用户明确要生成（生成吧/就这样/可以了）→ 重建后重跑 --preview，本轮回复＝最终分页卡片原样全文，最末单独一行输出 [[READY]]（只回标记不贴卡片＝把空输入交给渲染节点）
     · 主题要求含真实数据且 g_source 为空 → 意图卡加第 6 项「数据来源」，二选一：上传内部台账／暂缺留待补槽，绝不编造
  │（每份主对话各接一套就绪门 + 渲染，互不合流）
条件[就绪?A / 就绪?B]  {{ 主对话A.answer }} / {{ 主对话B.answer }} 包含 [[READY]]?
  ├ 是 → AI对话[渲染A / 渲染B]  Skill: create-ppt + download-link-publisher（两份同构）
  │        {{ 主对话A.answer }} / {{ 主对话B.answer }} 原文写入 preview.md → make_outline --from-preview 重建 outline.json
  │        → build_pptx → check_pptx → 发布 → 链接写进 answer（可选：变量赋值 g_done="yes"）
  └ 否 → 指定回复[等待A / 等待B]（自定义一句引导语，返回内容=开）→ 本轮结束，等用户下一条消息重跑
```

要点：

- **无循环**：迭代由用户发消息驱动，次数不限——彻底绕开「跳不出 / ≤9 / 停等」。
- **渲染确定化**：主对话平时不渲染；只有它吐出 `[[READY]]`，条件才放行到独立渲染节点，渲染只调一次，不误渲/重渲。
- **ELSE 必须以「返回内容=开」的指定回复收尾**：平台没有「结束」节点、条件分支不能悬空；若路径末节点返回内容=关，前端等不到终端回复、会话挂起不结束（实测）。指定回复内容是字面回复文本，写「结束当前会话」之类指令没有控制语义——只写一句简短引导即可。
- **两道门在对话里自然完成**：意图卡、大纲卡片都由主对话写进回复，用户直接聊天修改，比循环里「指定回复 + 判定 + 条件」省一大串节点。
- **卡片即数据，经 `--from-preview` 交渲染**：最终大纲以分页卡片形式落在生成轮主对话的 `answer` 里，渲染节点把 `answer` 原文写入 preview.md，跑 `make_outline.py --from-preview` **脚本化重建** outline.json——`src`/`caps` 由确定性重排还原、文本从卡片回填，全程无 LLM 改写；`[[READY]]` 与引导语被解析器自动忽略，重建自带 round-trip 自检。「重建」与「渲染」同在渲染节点完成，**不需要参数抽取节点**，也不需要 `g_outline` 全局变量，对话里任何轮次都没有 json 块。
- **卡片必须落在回复正文里**：`execute` 块的输出只有模型自己看得见，`answer` 只装模型写的文本。实测踩过——主对话跑完 `--preview` 却只回了引导语（大纲轮）、只回了 `[[READY]]`（生成轮），渲染节点拿到空输入后满世界找 `outline.json`、反复推理、最后追问用户，形成死循环。故三处同时加固：SKILL.md 5b 要求生成轮重贴卡片、系统角色点破「工具输出≠回复正文」、`--preview` 在 stderr 打同款提醒（走 stderr 才不会被复制进卡片）。
- **渲染节点不碰内容**：职责边界就是「重建 + 渲染 + 发布」。给它任何内容生成能力（联网、改写大纲），都等于授权它在拿不到大纲时自己编一份——客户确认的大纲与最终成品不再是同一份，比死循环更隐蔽。
- **直接文本/大纲**：靠 `g_source` 是否为空在主对话**内部**分流，不再单设 direct/topic 两条外部支路。素材段须用 `<素材开始>`/`<素材结束>` 围栏，否则空值塌缩会让主对话误判成直接文本路径、跳过门①。
- **文档提取覆盖 PDF（实测）**：所有文档类型（含 PDF）统一走「文档提取」`content`，取消原「PDF文档解析」分支与按扩展名分流，拓扑更简。
- **分支不合流 → 主对话复制两份**：平台多分支无法汇入同一节点，故「有上传/无上传」各接一份**同构**主对话（及各自的就绪门 + 渲染）。两份的系统角色/提示词/Skill 配置必须保持一致——改一处要同步另一处。上传素材经 `变量赋值` 落 `g_source`，主对话统一读 `{{ 全局变量.g_source }}`（无上传时为空），不出现 `{{ 文档提取.content }}` 这类跨节点直连引用。

### 5.4 节点逐项配置

**条件[上传分流]** — 变量 `{{ 开始.document }}`，两路：
- IF「有上传」：`不为空` → 文档提取 → 变量赋值 g_source → 主对话A。
- ELSE「无上传」：`为空` → 直接进主对话B。

两分支无法汇入同一节点，故各自续接一份主对话（配置见下，两份同构）。

**文档提取** — `选择文档 = {{ 开始.document }}`，输出 `content` → 变量赋值 `g_source ← {{ 文档提取.content }}`。**实测覆盖全部文档类型（含 PDF）**，无需再按扩展名分流、也不再挂「PDF文档解析」。仅在有新上传时运行；无上传的后续轮次 `g_source` 保留上一轮值（空则主对话走意图澄清）。

**AI 对话[主对话A / 主对话B]** — 全流程核心，**两份配置完全一致**（分支不合流才复制，改一处须同步另一处）。模型 `energy-deepseek-v3`（能编排 skill 又能写作）；**Skill 只挂 create-ppt**（不挂 download-link-publisher）；**历史聊天记录＝工作流、10 轮**；返回内容＝开；开启思考＝关。

⚠️ 系统角色里的 `{{ 变量 }}` 引用也会被平台按本轮值插值（实测无上传轮 `{{ 开始.document }}` 被替换成 `[]`，指令条件沦为乱码）——指令文本一律用文字指代提示词载荷字段（如「上传素材」段），变量引用只出现在提示词里。

系统角色：

```text
你是企业 PPT 生成智能体，内置 5 套设计模板，用 create-ppt 技能工作。与用户多轮对话，按两道确认门推进，绝不跳步直奔生成。

【模板解析】“初始模板”是起始单选的默认值。若对话历史中你已就换模板取得用户明确确认（如用户回复「切换到 X」），则以最近一次确认的模板为当前模板，忽略“初始模板”字面值；否则用初始模板。初始模板为空时，先让用户在 5 套里选定，或按主题推断并在卡片里明示请其确认——不得留空调用 --template。当前模板必须写入大纲 json 的 template 字段，并在意图卡/大纲卡中如实显示。切换确认后不要再重复门①的初始复述、也不要再次抛出同一切换提示。若素材/主题场景与当前模板明显不符，提示一行「您选了 X，但更像 Y，是否切换？回复『切换到 Y』即可，无需重开」。

【门① 意图】
- 提示词中「上传素材」段非空，或「用户消息」本身是成段文稿/已有大纲 → 直接文本路径，跳过意图澄清，直接进入门②。开头点明「已根据你上传/粘贴的素材直接生成大纲预览，如需先调整制作意图或页数请告知」。
- 否则 → 先澄清创作意图（制作目标 / 目标受众 / 页数规模 / 补充说明，缺失给默认值并标注），用编号卡片复述，让用户确认或修改；认可后进入门②。
- 「上传素材」段（<素材开始>/<素材结束> 之间）为空时，就是没有素材：不得声称「已根据你上传/粘贴的素材」。主题里提到「含事故数据」只是需求，不是素材。
- 主题要求含真实数据而上传素材为空时，意图卡追加第 6 项「6. 数据来源：您上传的内部台账／统计表（最准）」，可选值二选一：您上传的内部台账／统计表、暂缺（相关页留待补槽）。选定暂缺时数字位写「__」加极短待补说明（不用 XX/XXXX/{}），待补槽计入 caps，大纲卡后追加一行「以下页面含待补数据：第 N、M 页」。绝不联网检索、绝不编造具体数字。详见 skills/create-ppt/references/writing-guide.md 的「缺数据处置」。
- 意图卡末尾固定追加一行引导：「请回复『确认』继续，或直接提出修改（如『改成 12 页』『受众换成管理层』）；如需换模板，回复『切换到 X』。」

【门② 大纲】outline.json 是唯一事实源：骨架落盘 → 填充写盘 → 修改用 --patch 就地改盘 → --preview 读盘。
- 任何轮次都不输出大纲 JSON，回复只含分页卡片（--preview 原样输出，不改写、不节选）。
- 卡片只能是 --preview stdout 的原样复制，任何情况下不得手写、改写或补写卡片与「共 N 页」头行；脚本运行失败就如实报告失败原因，绝不输出形似卡片的内容——手写伪卡片会让后续轮次的重建必然失败。
- 卡片必须原样出现在你的**回复正文**里。工具执行（execute 块）的输出只有你自己看得见，用户和下游渲染节点都读不到——跑过 --preview 不等于回复里有卡片。漏贴＝大纲丢失，下游渲染节点会拿到空输入而无法出图。「以上为《标题》共 N 页大纲预览」这句引导语出现时，它上面必须真的有 N 页卡片。
- 首轮：uv run skills/create-ppt/scripts/make_outline.py --template <当前模板> --pages <页数> --title "<主题>" --sections "<章节>" > outline.json 生成骨架落盘，按 skills/create-ppt/references/writing-guide.md 填充 outline.json（要点条数不增删、每条不超该页 caps），uv run skills/create-ppt/scripts/make_outline.py --preview outline.json 输出分页卡片。本轮回复只含分页卡片预览（及末尾引导语）。
```
